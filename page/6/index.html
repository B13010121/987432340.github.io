反汇编库udis86的使用

发表于 2014-12-31   |   分类于 逆向分析

##解决2个BUG

GitHub项目地址，下载的是源代码，需要自己手动编译。百度了一下发现给出的帮助都是关于gcc下的，给跪了，叫我等常年使用VC6.0的情何以堪。

再去git上查看，找到了在windows平台上使用的相关介绍:

Build notes for Windows:
– If you don’t have python installed at C:\Python27, edit build.proj accordingly
– Execute buildVS2010.bat
– If you got VS2012, this will fail.
– Edit/Copy the build script, so that it finds vcvars32.bat (just needed to replace 10.0 with 11.0).
– Open the solution and update the toolset.
– Compile with the build script and ignore an error regarding ./Win32 not deletable.
– Be done with it

按照默认的操作：在pyhon2.7的环境下，安装vs2010，执行.bat，可是直接执行出现错误，一开始怀疑自己的python是不是版本不对，查看了一下是32位的python2.7，而VS是刚刚安装好的，应该不会有什么问题……

这时候打开 buildVS2010.bat,查看，发现作者把VS2010的路径写死在了C盘的默认路径上，像我一样不喜欢把程序安装在C盘上的人是不是都一样蛋疼啊！
再次执行，依然不对，bat文件中使用了 build.proj 文件，再打开查看，修改python的路径指向自己电脑的python路径，再次执行，终于成功了! 这样就多了一些 .h头文件出来。
接着，执行BuildVS2010\libudis86项目文件，然后编译生成一个udis86.lib文件。
再然后，将lib文件复制到VS2010安装目录的lib文件夹中，将udis86.h文件复制到VS2010安装目录的include文件夹中，下次使用以下代码即可使用udis86这个库:

linux下逆向分析小结

发表于 2014-11-12   |   分类于 逆向分析

因为要做一些CTF，里面的逆向大多是elf格式的，所以上github看了下国外逆向elf格式文件的一些个姿势，自己也实践了一下，记录有些分散，在这里把这些汇总串联起来。现在了解到的linux下的逆向有如下的几个方法：

  • strings 命令
  • ltrace 命令
  • EDG 动态调试
  • gdb 动态调试
  • IDA 远程调试

0x1 strings

strings 命令是linux命令行下的一个命令，功能是把指定文件中的可见字符串输出出来。个人理解，相当于OD下的字符串查找功能。用这个功能能把应用程序中的一些明文信息直接输出出来，有助于了解这个程序的一些提示信息（遇到password正确的提示就能很快的定位关键点了）。配合grep命令，例如grep 本次竞赛的flag格式的固定字符串有收获也说不定呢。

0x2 ltrace命令

和strings一样，也是linux下的一个命令，功能是把指定程序的函数调用流输出出来。他的原理大致如下：类似于调试运行这个程序，发现此程序调用了so中的函数时，将函数调用的参数，返回值都打印出来。和上面一样，利用grep命令效果更好，如果一不小心一些关键的函数尤其是字符串处理函数（例如strcmp strcpy等等）暴露了也能快速定位关键代码。

0x3 EDG

EDG是linux下的一个调试工具，模仿OD，作者说要做linux下的OD。所以界面 和 使用方面和OD差不多。在64位 ubuntu下安装这个工具的时候有点坑，总是提示插件路径不对，反复读作者的说明也未能实验成功，最后在google的帮助下找到了原因：64位的lib目录为 lib64 而作者把插件的路径写死成了lib 所以总提示中找不到插件。

0x4 gdb

大名鼎鼎的调试工具，不过在无源码的情况下的调试还是和有源码调试的时候不太一样，比如说我们不能直白的使用b main 这样的指令，因为没有符号信息在里面,为了便于调试，总结一些常见指令以备用。

  • 启动GDB 调试程序
    gdb ./xxx
  • 设置参数
    set args 参数1 参数2 ……
  • 设置反汇编语法为intel （我个人还是习惯看这种语法的反汇编代码）
    set disassembly-flavor intel
  • 查看入口点函数地址
    info file （记下入口点的地址）
  • 在入口点设置断点
    b *入口点地址
  • 让程序跑起来
    r
  • 查看当前位置20行的反汇编代码
    x/20i $pc
  • 查看断点
    info b
  • 查看当前寄存器的值
    info r
  • 查看当前内存的值
    x /nfu
    • n表示要显示基本块的个数
    • f表示对基本块的解释方式, 可取如下值
      • x 按十六进制格式显示变量。
      • d 按十进制格式显示变量。
      • u 按十进制格式显示无符号整型。
      • o 按八进制格式显示变量。
      • t 按二进制格式显示变量。
      • a 按十六进制格式显示变量。
      • i 指令地址格式
      • c 按字符格式显示变量。
      • f 按浮点数格式显示变量。
    • u表示一个基本块的占几个字节
      • b表示单字节，
      • h表示双字节，
      • w表示四字节，
      • g表示八字节

比如：x/3uh 3个基本块，每个块都是双字，按照16进制格式显示

0x5 IDA

IDA调试的用法我比较倾向于在win下调试虚拟机中的elf文件，因为win下的ida是破解过的高版本，有比较好的库识别。

##0x1 babyCrack

不得不吐槽一下出题的哥们把题目写错了，只要第一个参数满足条件，就会提示正确，唯一能看到的明码字符串提交上去也不正确（具体记不清了）～一开始还以为有什么特殊的技巧，因为看着已经有人做出来了……还好没多久，主办方修改了key，终于提交成功～

##0x2 babyCrack2

这题是个.NET程序，用大名鼎鼎的.NET逆向工具打开，找到程序的反编译出来的源码即可得到明文字符串的key。最多考察一下相关工具的使用吧。

##0x3 wow

这题是这次比赛收获比较大的一道linux逆向题目，帮助我练习了一下linux下的逆向，然后还有一点点小trick的积累。首先是运行，发现直接跑是跑不起来的，发现so文件缺失，心里想着先用IDA直接反编译吧。反编译之后，发现可以根据残留的函数命名找到关键的函数位置。来到算法的地方，经过一番分析之后，抽象出来这实际上是一道数学问题– 解n元n次方程组的问题。我似乎想起了线代老师倩丽的身影，可是缺想不到怎么去解这个方程。学渣的心酸吖。正准备45度仰望星空感叹的时候，扫见了桌上的python书，记得借python书的时候旁边有本书写python用于科学计算的。于是搜了搜python有没有解N元N次方程的题目的库，没想到百度百科上居然都有，我是有多落伍……果断把numpy这个库下下来，只需一秒，结果马上出来了。又一次感慨搞算法的人的强大之处。

##0x4 总结

终于结束了，后半场都没有怎么输出（回去睡觉了）…… 逆向题目该能做的都做出来了，关于APK的题目目前还是两眼一抹瞎，移动端可是未来的主流，而android是未来的windows，所以移动安全真的很重要，所以该好好关注一下安卓逆向了（IOS也买不起吖）。然后就是感觉在一次比赛中能有所收获，有经验的积累自己就很满足了。尽管认识到和高手们的差距还是很大。不过古语云：不积跬步，无以致千里。一点一滴的扎实的积累才是真正的成长！
